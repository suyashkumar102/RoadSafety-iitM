# Winning Plan: Choose RoadSoS for a 3-Member Team

## Summary
- **Recommended problem statement:** **RoadSoS**.
- **Why:** Best reward per unit risk. It has a strong live-demo emotional hook, measurable lifesaving relevance, achievable data collection, and a clearer path to offline functionality than DriveLegal or RoadWatch.
- **Source basis:** Official event page confirms the 2026 hackathon window and extension to May 31 ([CoERS/RBG](https://coers.iitm.ac.in/event/national-road-safety-hackathon-2026/)); Unstop confirms one submission, code, 7-slide deck, assumptions/package document, and the three tracks ([Unstop](https://unstop.com/hackathons/national-road-safety-hackathon-2026-iit-madras-1656937/amp)); CoERS AI summit context emphasizes scalable, data-driven, governance-compatible road safety systems ([CoERS AI Summit](https://coers.iitm.ac.in/event/ai-for-road-safety-summit/)); India emergency fallback should use official ERSS 112 guidance ([MHA ERSS](https://www.mha.gov.in/en/commoncontent/emergency-response-support-system-erss)).
- **Important note:** No public point-weighted scoring rubric was found. This plan uses the published evaluation criteria as the official rubric and adds an internal 100-point optimization proxy.

## Problem Selection
| Track | Risk | Reward | Common Weakness | Verdict |
|---|---:|---:|---|---|
| DriveLegal | High | Medium-High | Legal hallucination, stale rules, jurisdiction complexity | Avoid |
| RoadWatch | Very High | High | Hard public works data, unclear complaint routing, many pothole apps | Avoid |
| RoadSoS | Medium | Very High | Data reliability and offline quality | Choose |

- **RoadSoS winning angle:** Build an **offline-first accident response orchestrator**, not a generic chatbot.
- **Core promise:** “In the golden hour, a bystander can find the right help in under 10 seconds, even with poor network.”
- **In scope:** Nearby trauma/hospital/police/ambulance/tow contacts, ranked by distance and reliability; emergency fallback; offline cache; incident summary; multilingual/low-literacy UI; source-backed data.
- **Out of scope:** Real ambulance dispatch, diagnosis, hospital bed availability, guaranteed live contact verification, legal/medical decision-making.

## Judge Strategy
- **Judge pain points to solve:**
  - Too many submissions will be thin ChatGPT wrappers.
  - Emergency tools that fail offline are not credible.
  - Fake or unsourced contacts destroy trust.
  - Pretty UI without source provenance will feel unserious.
  - “Global applicability” often becomes a vague claim instead of an architecture proof.
- **Seen-before approaches to beat:**
  - “Call 112/108” chatbot with no location ranking.
  - Google Maps search wrapper.
  - Static emergency directory.
  - Landing page plus basic chatbot.
  - AI-generated advice without guardrails.
- **Judge delight triggers:**
  - Open demo in airplane mode and still fetch nearby emergency contacts.
  - Show source badges, distance, last-verified date, and confidence score.
  - One-tap emergency flow: detect/manual location -> ranked contacts -> call/share incident summary.
  - “No hallucinated contacts” rule visible in the demo.
  - A 30-second staged accident scenario that works smoothly from IIT Madras/Chennai.
  - A second-country sample dataset proving the global schema, even if coverage is limited.

## Product Architecture
- **Frontend:** Mobile-first React/Vite PWA.
- **Backend:** Python FastAPI.
- **Database:** SQLite generated from curated CSV/JSON seed files.
- **Offline:** Service worker plus IndexedDB/static JSON cache for demo city emergency contacts.
- **AI layer:** Guarded assistant for summarizing incident info, translating instructions, and answering “what should I do next?” using only approved templates and retrieved data.
- **Geospatial logic:** Deterministic ranking first: distance, service type, source reliability, verification freshness, availability, and emergency priority.
- **Emergency fallback:** Always show official national emergency fallback from the dataset; for India, include ERSS 112 as the primary fallback.

## Public Interfaces and Data Contracts
- `POST /api/nearby-services`: accepts `lat`, `lon`, `radius_km`, `service_types`; returns ranked emergency services with source and confidence metadata.
- `POST /api/incident-summary`: accepts location, injuries count, vehicle status, notes; returns a concise shareable responder summary.
- `POST /api/assistant`: accepts user message and location context; returns guarded response with citations to local data or approved safety templates.
- `GET /api/cache-package`: returns offline emergency dataset version for the frontend.
- Contact schema must include: `id`, `name`, `type`, `lat`, `lon`, `phone`, `address`, `locality`, `country`, `source_url`, `verified_at`, `availability`, `confidence_score`, `notes`.

## Fixed Team Constraints
- No AI-generated phone numbers, addresses, hospital names, police stations, or ambulance contacts.
- Every displayed service must come from the dataset and show source/verification metadata.
- Offline mode must work for at least one polished demo region: **IIT Madras/Chennai**.
- Add one lightweight second-region sample, preferably another Indian city or BIMSTEC city, to prove portability.
- First screen must be the emergency action screen, not a marketing landing page.
- No secrets in repo; use `.env.example`.
- All shared behavior goes through API/data contracts before UI work depends on it.
- The final submission must include code, 7-slide presentation, assumptions/package document, and dataset notes.

## Team Split
- **Member 1: Data + Geo Reliability**
  - Owns emergency dataset, schema, dedupe, geocoding, source documentation, confidence scoring, and ranking tests.
  - Delivers Chennai seed data plus one portability sample region.
- **Member 2: Backend + AI Guardrails**
  - Owns FastAPI, SQLite integration, `/nearby-services`, `/incident-summary`, `/assistant`, no-hallucination guardrails, and API tests.
  - Ensures all assistant responses cite dataset/template sources or say it cannot verify.
- **Member 3: Frontend + Demo + Submission**
  - Owns PWA UI, offline cache, map/list view, one-tap call/share actions, accessibility, slide deck, README, and final demo script.
  - Ensures the app feels usable under stress on a phone-sized viewport.

- `data/`: shared curated datasets and generated SQLite file.
- `backend/`: API implementation; only Member 2 merges backend changes unless coordinated.
- `frontend/`: PWA implementation; only Member 3 merges UI changes unless coordinated.
- `docs/`: assumptions, package list, data sources, submission document, and slide outline.
- `demo/`: golden demo scenario, screenshots, test coordinates, and final pitch script.
- Daily merge rule: feature branches merge into `main` only after the golden demo scenario still passes.

- Offline functionality: 20 points.
- Innovation and additional features: 15 points.
- Cross-region/cross-country architecture: 10 points.
- UI clarity and accessibility: 5 points.
